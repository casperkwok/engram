# Mneme — 面向 LLM 智能体的记忆格式

[English](SPEC.md) · **中文**


**版本 v0.1 · 草案** · © 2026 Casper Kwok · Apache-2.0

**一句话定义**：Mneme 是一种纯文本格式，用来存放智能体（agent）在工作过程中产生的、需要跨会话保留的**持久记忆**——决策、事实、偏好、踩坑、目标——并让概率性的读者（大模型）能低成本、可信任地读写它。

文件扩展名：`.mneme`（短写 `.mn`）。它物理上是带约定的纯文本，任何编辑器、任何模型都能直接打开；它的"格式"不在符号，而在下面这套被记忆这件事的物理约束逼出来的结构。

---

## 1. 它是什么

Mneme 不是文档格式，也不是数据库。文档为"被人阅读/渲染"服务，数据库为"被程序精确查询"服务。Mneme 为第三种东西服务：**一个概率性读者（模型）需要反复读取、并据此行动的长期记忆**。

它的世界由两层构成：

- **Cell（记忆细胞）**：记忆的原子单位。一条决策、一个事实、一项偏好，就是一个 cell。每个 cell 自带身份、要旨、生命周期状态、置信度、新鲜度、召回线索和正文，**能脱离上下文独立被理解**。
- **Spine（脊柱）**：所有 cell 的极简投影——每个 cell 抽出一行（id + 状态 + 要旨）。脊柱**不是手写的，是从 cells 生成的**。它是智能体"常驻在工作记忆里"的那份索引：廉价、可一眼扫完、随时知道"我都知道些什么"。

智能体不把整个记忆库灌进上下文，而是：常驻脊柱 → 按需把相关的几个 cell 展开。这是整个设计的运转核心。

---

## 2. 为什么这样设计

每一个设计决定都对应 LLM 记忆这件事的一条物理约束。这些约束，`html` / `md` / `org` / `json` 都没有正面回应过——因为它们的读者是人或确定性程序，不是模型。

| 约束（LLM 记忆特有） | 逼出来的设计 |
|---|---|
| **读者是概率性的**，靠模式重建意义，孤立片段易误读 | 每个 cell 自带身份、状态、要旨，可脱离上下文自解释；冗余在这里不是浪费，是纠错 |
| **读贵写便宜**：每次读都烧 token，写只发生一次 | 两层结构。常驻廉价的 spine，按 `cue` 命中才展开完整 cell；记忆库再大，进上下文的也只有脊柱 + 几个相关 cell |
| **记忆会过期、被推翻、会衰减**；文档是静止的，记忆不是 | `state` / `conf` / `seen` 成为一等公民——格式记录的不只是内容，还有**这条记忆与"当下现实"的关系** |
| **召回是语义联想的**，不靠路径 | `cue` 行：每个 cell 自报"聊到这些话题时该把我捞出来"，让格式配合检索而非对抗它 |
| **写它的也是模型**：模型怕数括号、怕深层缩进，不怕语义标签 | line-oriented 语法，一行一个语义键，不嵌套、不依赖精确缩进，坏了也容易自愈 |
| **记忆需要可审计**：为什么当初这么定，不能丢 | 只追加、不修改内容；要改就新增 cell 并 `supersede`，旧的连同它的 `why` 一起留存 |

其中最关键、也最区别于一切文档格式的一条原则：

> **内容不可变，状态可变。**
>
> 一个 cell 的 `gist` / 正文 / `cue` 记录的是"在某个时刻知道/决定了什么"——这是关于过去的事实，永不更改。
> 而 `state` / `seen` / `conf` 记录的是"这条记忆与现在的关系"——会随现实推移而更新。
> 想改内容？新建一个 cell 去取代旧的。这条规则同时满足了"可审计"（历史不丢）和"可保鲜"（状态能更新）这对看似矛盾的需求。

## 3. 格式规格

### 3.1 一个 Cell 的解剖

```
@ DEC-0042  storage/cache
gist  本地缓存用 SQLite 取代 JSON 文件
state live   conf high   since 2026-06-03   seen 2026-06-03
cue   并发写 / 缓存损坏 / 查询慢 / 本地存储选型
> JSON 并发写会损坏，查询得全量加载。改用 SQLite：单文件、事务、增量查询。
> 代价：多一个依赖，要写迁移脚本。
link  supersedes DEC-0031
```

cell 之间用一个空行分隔。一个 cell 以 `@` 行开始，到下一个 `@` 行或文件结束为止。

### 3.2 字段参考

| 行首 | 含义 | 必填 | 说明 |
|---|---|---|---|
| `@ <ID>  <topic-path>` | cell 头：身份 + 主题路径 | 是 | `ID` 形如 `PREFIX-NNNN`，全局唯一、永不更改、永不复用。`topic-path` 是 `/` 分隔的层级（如 `storage/cache`），用于人眼扫读和粗筛 |
| `gist` | 一句话要旨 | 是 | 单行，记忆的精华。**会被提升进 spine**。必须自包含——单独拎出来也读得懂 |
| `state` | 生命周期状态 | 是 | 见 §3.3 |
| `conf` | 置信度 | 否 | `high` / `med` / `low`，缺省为 `med`。表示"我多确定" |
| `since` | 确立日期 | 是 | 这条记忆首次确立的日期 |
| `seen` | 最近确认日期 | 是 | 上次被复核/重申仍成立的日期。**衰减信号** |
| `cue` | 召回线索 | 推荐 | `/` 或 `,` 分隔的话题/关键词。智能体据此判断"当前任务该不该把我捞出来" |
| `>` | 正文 | 否 | 自由散文，写理由、细节、上下文。可多行 |
| `link` | 类型化关系 | 否 | `link <关系> <ID>`，可重复。关系见 §3.4 |

`state conf since seen` 习惯写在同一行，以 `键 值` 空格对的形式排开，便于人眼一眼读完状态。也可拆成多行，解析不受影响。

### 3.3 状态与生命周期

| state | 含义 | 是否进默认工作记忆 |
|---|---|---|
| `proposed` | 提议中，尚未成为权威结论 | 否（除非任务正是在评估它） |
| `live` | 当前成立、生效中。默认的权威态 | 是 |
| `stale` | 可能过期、久未复核；依赖前需先核实 | 是（但带警惕标记） |
| `superseded` | 已被更新的 cell 取代，仅作历史保留 | 否 |
| `retired` | 彻底失效（关闭的 OKR、放弃的方案），仅作审计保留 | 否 |

允许的流转：`proposed → live`；`live ⇄ stale`；`live / stale → superseded`（被取代）或 `retired`（废弃）。
内容（gist / 正文 / cue）一旦写下不再更改；流转只动状态行。

### 3.4 关系类型（`link`）

`supersedes` / `superseded-by`（成对）、`relates`（相关）、`depends-on`（依赖）、`blocks`（阻塞）、`refines`（细化）。取代关系应成对标注：新 cell 写 `link supersedes OLD`，旧 cell 写 `link superseded-by NEW`。

### 3.5 ID 前缀约定（可扩展）

| 前缀 | 类型 |
|---|---|
| `DEC` | 决策（decision） |
| `OKR` | 目标与关键结果 |
| `PREF` | 用户/项目偏好 |
| `GOTCHA` | 踩坑/陷阱/教训 |
| `FACT` | 事实/约束/环境信息 |
| `TODO` | 待办（注意：长期未决才入库，临时待办别放这） |

需要新类型就加新前缀，约定本身是开放的。

### 3.6 Spine（脊柱）的定义

脊柱是 cells 的**投影**，不手写。生成规则：对每个 cell 输出一行 `<ID>  <state>  <gist>  [topic-path]`；默认只投影 `state ∈ {live, stale}` 的 cell（`proposed/superseded/retired` 不进工作记忆）。

```
DEC-0042   live   本地缓存用 SQLite 取代 JSON 文件        [storage/cache]
PREF-0007  live   提交信息用中文，首行 ≤ 30 字            [style/commit]
GOTCHA-003 stale  迁移脚本在 Windows 路径下会失败          [storage/migration]
```

生成方式：一次 grep（抓 `@` / `gist` / `state` 行）或 `tools/mneme_tool.py spine`。**单一真相源永远是 cells，脊柱永远是派生物**——不存在双份记账。

### 3.7 非正式文法

```
file      := comment* cell (BLANKLINE cell)*
comment   := "#" TEXT NL                       # 以 # 开头的行被忽略
cell      := header gistline statusline cueline? bodyline* linkline*
header    := "@" SP ID SP+ topicpath NL
gistline  := "gist" SP+ TEXT NL
statusline:= ("state" SP+ STATE | "conf" SP+ CONF | "since" SP+ DATE | "seen" SP+ DATE)+  (同行或多行)
cueline   := "cue" SP+ TEXT NL
bodyline  := ">" SP TEXT NL
linkline  := "link" SP+ REL SP+ ID NL
```

解析容错原则：行首语义键是锚点；未识别的行忽略而非报错；缺省字段取默认值。这是为了让"模型写出来的不完美文本"也能被稳健消费。

## 4. 智能体如何使用 Mneme（核心）

以下是 agent 在运行时应当遵循的协议。把它们理解为智能体面对记忆库时的标准操作流程。简短版见 [AGENT.md](AGENT.md)（可直接粘贴进系统提示）。

**协议 A —— 启动定向（Boot）**：会话开始、或任务涉及项目背景时，**先加载 spine，不要读完整文件**。把脊柱当作"我都知道些什么"的轻量索引持有在上下文里。

**协议 B —— 召回展开（Recall）**：动手前，拿当前任务的话题去匹配脊柱里的 `gist`、各 cell 的 `cue` 和 `topic-path`。**只展开命中的那几个 cell**。不要把整个记忆库倒进上下文。命中标准宁可宽一点（召回优先）。

**协议 C —— 信任校准（Trust）**：依赖一条召回的 cell 前看它的状态：
- `live` + `conf high` + `seen` 较新 → 直接信任使用。
- `stale`、或 `conf low`、或 `seen` 已很久 → **不要当然相信**。先核实；通过后把 `seen` 更新为今天（必要时 `state` 拨回 `live`、`conf` 上调）。
- `proposed` → 待定项，非结论；除非任务就是在推进它，否则不据此行动。

**协议 D —— 写入新记忆（Write）**：当本次工作产生了一条**未来会话应当知道**的持久信息，就新建一个 cell：
1. 取下一个 id（建议用 `mneme_tool.py new-id PREFIX`，防撞号）。
2. 写一句**自包含**的 `gist`。
3. 设 `state`（刚敲定→`live`；斟酌中→`proposed`），`conf`，`since = seen = 今天`。
4. 写 `cue`：想清楚"未来什么任务话题下该重新捞出它"。这一步决定将来能否被找到，别省。
5. 用 `>` 写正文：理由、代价、上下文。
6. 有关联就加 `link`。
7. **追加到文件末尾**。绝不顺手改无关旧 cell。

判断准绳：**这条信息下次新开会话时，没有它会让我做错或重复劳动吗？** 是→写。

**协议 E —— 更新与取代（Supersede）**：修正已有认知时——**不要编辑旧 cell 内容**。新建带新内容的 cell；旧 cell 置 `state superseded` 加 `link superseded-by 新ID`；新 cell 加 `link supersedes 旧ID`。若只是重申仍成立 → 把 `seen` 更新为今天（唯一允许的对既有 cell 的改动，且改的是状态非内容）。

**协议 F —— 衰减巡检（Decay sweep）**：`seen` 超时限且未被重申的 `live` cell → 置 `stale`；明确不再相关的 → `retired`。建议交给工具/钩子，别靠自觉（见 §6）。

**协议 G —— 冲突处理（Conflict）**：两条 `live` 矛盾时优先 `conf` 高者，其次 `seen` 新者；**并显式告诉用户**，而非默默选一个。处理后把落败方 `supersede`。

**协议 H —— 卫生（Hygiene）**：一 cell 一主张；`gist` 自包含；长内容（完整文档、长代码）外置，cell 里 `link` 或引用路径；id 一经分配永不变、不复用。

## 6. 工具与钩子（让格式不靠 agent 自觉）

`.mneme` 是纯文本，但有两类账目**靠概率性读者自己维护必然腐化**，应交给确定性工具：

- **ID 分配**：手数"下一个 id"会在多写入端撞号。用 `tools/mneme_tool.py new-id PREFIX` 确定性分配。
- **生命周期新鲜度（`seen`/`state`）**：模型不会主动做衰减巡检。用 `tools/mneme_tool.py lint` 标记 `seen` 超期（默认 >45 天）的 live cell，并校验 supersede 链对称、悬空链、重复 ID、非法日期。

`mneme_tool.py`（零依赖，stdlib）：

```bash
python tools/mneme_tool.py spine  <file>        # 派生脊柱（live/stale）
python tools/mneme_tool.py lint   <file>        # 体检；有 error 退出码 1（供钩子用）
python tools/mneme_tool.py new-id DEC <file>     # 分配下一个唯一 ID
```

**pre-commit 钩子**：把 `lint` 挂进 `.git/hooks/pre-commit`，提交涉及 `.mneme` 时自动跑——结构错误挡提交、衰减/不对称仅告警。这样**腐化每次提交可见，而非默默烂掉**。Mneme 的精神就是"概率性读者不可信"——那记账更不能靠它自觉，交给机器把关。

## 7. 人类如何写

人可以直接手写或编辑 cell——语法就是为"人也能轻松读写"设计的。扫脊柱掌握全局，细看某个 cell 时正文是 `>` 的人话。唯一要守的纪律和智能体一样：**改认知靠新增+取代，不靠覆盖**；重申只动 `seen`。

## 8. 非目标与适用边界

**适合**：长生命周期、多会话的 agent 记忆；跨项目/常驻 agent 的累积记忆；需要审计"当初为什么"的团队；多人/多 agent 共享的长期记忆（先解决 ID 分配）。

**不适合**：
- **不是数据库**：不为高频精确查询、事务、大规模联表设计。
- **不存大块内容**：长文档、完整代码放别处，cell 里只留要旨 + 链接。
- **不是临时草稿**：只装"跨会话值得记住"的蒸馏物。
- **不替代代码与文档**：它存"为什么"（决策、约束、教训、偏好），而非"是什么"的全部细节。
- 小型短期项目：一个带状态字段的 markdown 列表往往就够了，未必需要本格式。

## 9. 已知局限（v0.1）

- **P3 · 就地改状态行**：`seen`/`state` 仍需就地编辑，模型可能改错行。`lint` 会兜底发现结构损坏；更稳的解法（状态变更走事件追加、读取时折叠）留待后续版本。
- **真正并发多写入端**：sequential ID（`DEC-0043`）在并发写时仍可能竞争；那种场景改用随机后缀 ID（`PREFIX-<rand>`）或按写入端分前缀命名空间。
- 这些局限本身建议用一条 `FACT` cell 记进项目的元记忆，吃自己的狗粮。
